# WBS — MemoX v1 (local-first Flutter app)

Full Work Breakdown Structure for MemoX v1: a **local-first** flashcard / SRS
app. Flutter owns everything — UI, state, domain logic, and local persistence.
There is **no remote server in v1** ("BE" here = the app's on-device domain +
data layers). Built so the **FE track** (design-system → components → screens)
and the **BE track** (domain → SRS engine → Drift data) proceed **in parallel**
once shared foundations land.

Confirmed decisions (v1) — summary; full rules in
[`docs/business/`](../business/index.md):
8-box single-direction SRS · 20 new cards/day · nested decks · search term+meaning
· soft-dup · goal + streak · Premium/account-sync **deferred**. Stack: **Riverpod
Annotation** (state + DI) · **Drift/SQLite** (persistence) · **go_router** ·
Material 3 · design tokens already generated (Tier 0).

> **Loop execution.** Every task has a self-contained loop prompt under
> [`docs/agent/build/`](../agent/build/README.md) (generated by
> `tool/design/gen_task_prompts.mjs`). Run with `/loop`; the queue picks the next
> pending task by phase + deps.

## Architecture (per `flutter-full-app` skill)

```
lib/
├── app/                         # bootstrap: ProviderScope, MaterialApp, env/flavor
├── core/
│   ├── constants/  error/  logging/  utils/
│   ├── routes/                  # go_router (typed routes, shell + bottom nav)
│   └── theme/                   # Tier-0 tokens (generated) + app_theme + mx_theme
├── domain/                      # ── BE core (pure Dart, no Flutter/Drift) ──
│   ├── entities/                # Deck, Card, ReviewLog, StudySession, Goal…
│   ├── repositories/            # abstract interfaces
│   └── usecases/                # SRS engine + per-capability use cases
├── data/                        # ── BE impl ──
│   ├── datasources/local/       # Drift DB, DAOs
│   ├── models/                  # Drift tables + row↔entity mappers
│   └── repositories/            # repository implementations
├── presentation/
│   ├── shared/
│   │   ├── primitives/          # Mx atoms (Button, Chip, Avatar…)
│   │   ├── composites/          # Mx composites (Card, Scaffold, AppBar, Nav…)
│   │   └── layouts/  screens/   # app-wide reusable shells
│   └── features/<feature>/
│       ├── providers/           # @riverpod notifiers (own mutation)
│       ├── screens/             # render + dispatch intent
│       └── widgets/             # feature-local components
└── l10n/                        # ARB + generated AppLocalizations
```

**Layer contracts** (enforced): foundation is context-free/token-only ·
primitives wrap Material, no business logic / provider / feature imports ·
composites compose primitives, feature-independent · feature UI may own provider
wiring but must not import repositories/datasources/`dart:io` · dependency
direction one-way `Feature → Composite → Primitive → Core`.

## Tracks & parallelism

```
                     ┌──────────────────────────────────────────────┐
  Phase I (infra) ──►│ Phase T (theme)      Phase DM (domain core)   │  ← shared foundation
                     └───────┬───────────────────────┬──────────────┘
                             │ (contracts land)       │
             FE TRACK ◄──────┘                        └──────► BE TRACK
        P primitives → K composites → S screens   DT data (Drift) impls + wire usecases
                             │                        │
                             └────────► Phase V ◄─────┘  (integration + goldens)
```

The seam is **Phase DM** (domain contracts: entities + repository interfaces +
usecase signatures). Once it lands, screens code against usecase interfaces +
providers (backed by in-memory fakes until DT lands), while the BE track
implements Drift + real repositories. Neither blocks the other.

## Conventions

- **ID**: `<phase>.<n>`. **Status**: ☐ · ◐ · ☑ · ⊘ deferred. **Size**: S/M/L.
- **Source of truth**: the kit is frozen (design); v1 product rules live in
  **`docs/business/`** (BRD-lite `index.md` + per-feature specs) — the DM/domain
  tasks read them. Never edit the kit to fit Flutter. If a rule is missing, STOP
  and confirm — don't invent.
- **Codegen**: Riverpod + Drift use `build_runner`. Generated `*.g.dart` /
  `*.drift.dart` are build outputs; a CI gate checks they're up to date.
- **Strings** from ARB, never hardcoded.

### Definition of Done (per task; track-specific tests)

1. **Built** at the mapped path, respecting layer contracts.
2. **Analyzes** — `dart analyze lib test` → 0 issues; codegen up to date.
3. **Tested** — the right pyramid level: **domain** = pure unit tests
   (deterministic, no Flutter) · **data** = Drift integration tests (in-memory DB)
   · **primitives/composites** = widget + golden (light+dark) · **screens** =
   widget tests over provider states + golden vs `shots/*.png`.
4. **Parity / correctness** — UI matches the kit for every state; domain matches
   the v1 rules in `docs/business/` with edge cases covered.
5. **Decision Table** — every `D-xxx` row this task touches
   ([`core-decision-table.md`](../decision-tables/core-decision-table.md)) has a
   covering test; cite the `D-xxx` id(s) in the Ledger. See §Decision trace.
6. **Ledger** — row(s) in §Ledger (kit/`D-xxx` node → Dart symbol → test).
7. **Gates green** — `node tool/verify/run.mjs` passes.

### Verify

The single gate is **`tool/verify/run.mjs`** (created in **I.0**); tasks call it,
not the raw commands.

```bash
node tool/verify/run.mjs          # full: codegen freshness + gen_tokens --check + analyze + test
node tool/verify/run.mjs --quick  # analyze + test only
node tool/verify/run.mjs --docs   # doc/spec freshness + gen_tokens --check only
```

Before I.0 lands, fall back to:
`dart run build_runner build --delete-conflicting-outputs && node tool/design/gen_tokens.mjs --check && dart analyze lib test && flutter test`.

## Critical path

```
I.0→I.1→I.2→I.3→{I.4,I.5,I.6,I.7,I.8}→I.10→I.9 (foundation seal) ─┐
   ├─► T.1→T.3→T.4 ─────────────────► P.* → K.* → H.* → H.08 gate ─┤
   ├─► DM.1→DM.2→DM.3/DM.8→DM.9 (fakes) ───────► S.00 → S.01 pilot ┼─► V.*
   └─► DM.4 (SRS) ─► DT.0→DT.0.1→DT.1→DT.4→DT.5/DT.7 (Drift) ───────┘
```

**Two hard gates**: **I.9** seals "foundation usable" — nothing in Phase S/features
starts before it. **H.08** seals the shared UI layer before the 22 screens. **DM.9**
fakes let the FE track run before Drift (DT) exists.

Bottlenecks: **I.0** (the gate every task calls) · **I.9** (foundation seal) ·
**T.1** (all UI needs it) · **DM.2/DM.3** (contracts unblock both tracks) · **DM.4**
(SRS) · **DT.0/DT.0.1** (schema + safety gate all Drift) · **H.08** (gallery gate).

### Recommended execution order

`I.0 → I.1,I.2,I.3 → I.4,I.5,I.6,I.7,I.8 → I.10 → I.9` · then `T.1–T.6` · then
`DM.1–DM.3, DM.8, DM.9` · then `P.* → K.* → H.* → H.08` · then `DT.0, DT.0.1` ·
then `DM.4–DM.7` + `S.00` → **S.01 dashboard pilot** (review) → fan out S/DT/V.

## Risks

- **R1 — Golden brittleness** across fonts/platforms → load kit font in tests;
  canonical host; tolerances.
- **R2 — SRS correctness** — the 8-box scheduler is the product; a bug is silent
  data damage → exhaustive unit tests before any screen consumes it (DM.4 gates S).
- **R3 — Drift migrations** — schema drift corrupts user data → migration tests
  from v1 forward, never edit a shipped schema in place.
- **R4 — FE/BE contract drift** — if DM interfaces change after screens code
  against them → treat DM.3 as a frozen contract; version changes explicitly.
- **R5 — Feature-local scope creep** (65 local components) → pilot dashboard,
  size-check each screen.
- **R6 — Undrivable kit states** (error/loading behind a Result-based notifier)
  → document as gaps in the Ledger, don't fake.
- **R7 — Kit drift mid-build** → `gen_tokens --check` surfaces it; regenerate.
- **R8 — Decision coverage** — a behaviour rule with no test silently regresses
  → every `D-xxx` maps to a task+test (§Decision trace); DoD enforces it.
